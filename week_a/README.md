## 3. Lab Overview

This lab demonstrates the ability to manually deploy a Google Compute Engine (GCE) virtual machine in GCP that serves a live web page on port 80. The VM runs an nginx web server with a custom SEIR-I Ops Panel homepage. The deployment is validated using a gate script that proves the infrastructure is reachable, healthy, and self-identifying.

This lab establishes the foundational principle of SEIR-I:

> "Real engineers never say 'it works on my screen.' They prove it."

## 6. Steps to Complete the Lab

### Step 1 — Set up GCP Project
- Log into GCP Console at console.cloud.google.com
- Create or select your project
- Enable the Compute Engine API

### Step 2 — Initialize gcloud CLI
```bash
gcloud init
gcloud config set project YOUR-PROJECT-ID
gcloud auth application-default login
```

### Step 3 — Create the VM
- Navigate to Compute Engine → VM Instances
- Click Create Instance
- Set machine type to e2-micro (free tier eligible)
- Set region to us-central1-a
- Under Firewall — check Allow HTTP traffic (port 80)
- Under Advanced → Management → Startup Script — paste the supera.sh startup script
- Click Create

### Step 4 — Wait for VM to Start
- Wait 2-3 minutes for the startup script to complete
- nginx installs and configures automatically via the startup script

### Step 5 — Verify the Deployment

**Browser proof:**
```
http://YOUR-EXTERNAL-IP/
```

**SSH into the VM and curl locally:**
```bash
curl localhost
curl -s localhost | head
```

**Service proof:**
```bash
systemctl status nginx --no-pager
```

**Machine proof:**
```bash
curl -s localhost/healthz
```

**Engineer proof:**
```bash
curl -s localhost/metadata | jq .
```

### Step 6 — Run the Gate Script
```bash
chmod +x gate_gcp_vm_http_ok.sh
VM_IP=YOUR-EXTERNAL-IP ./gate_gcp_vm_http_ok.sh
```

Gate script source:
https://github.com/BalericaAI/SEIR-1/blob/main/weekly_lessons/weeka/script/gate_gcp_vm_http_ok.sh

### Step 7 — Collect Artifacts
- `gate_result.json` — generated by the gate script
- `badge.txt` — generated by the gate script
- Screenshot of homepage in browser

## 8. Teardown

To stop the VM and avoid charges:

1. Go to GCP Console → Compute Engine → VM Instances
2. Select your VM
3. Click **Stop** — this pauses billing for compute
4. The boot disk continues to incur a small storage charge (~$0.04/GB/month)

To fully destroy the VM:
1. Select the VM
2. Click **Delete**
3. Confirm deletion — this removes the VM and the boot disk

> **Note:** Always stop or delete your VM after completing the lab to avoid unexpected charges.

## 9. Lessons Learned

### What is relatable to the user/customer?
Deploying a VM that serves a live webpage demonstrates the same process used to host real applications in the cloud. The startup script pattern mirrors how production servers are bootstrapped automatically without manual intervention after launch.

### Struggles encountered
- The startup script requires time to complete after the VM starts. Attempting to access the homepage before nginx finishes installing results in a connection refused error. Patience and `systemctl status nginx` are required to confirm readiness.
- Finding the correct external IP each time the VM restarts — ephemeral IPs change on every start/stop cycle.
- The gate script must be made executable with `chmod +x` before it can run.

### How did you save money after teardown?
- Stopped the VM immediately after completing the lab and collecting artifacts
- Verified no other resources (NAT Gateways, Load Balancers) were left running
- Monitored GCP Billing → Budgets & Alerts for unexpected charges

## 11. Troubleshooting

### VM is running but homepage is not loading
```bash
systemctl status nginx --no-pager
journalctl -u nginx --no-pager
sudo systemctl restart nginx
```

### Gate script fails on /healthz or /metadata
```bash
sudo nginx -t
curl -v localhost/healthz
curl -v localhost/metadata
```

### Permission denied running gate script
```bash
chmod +x gate_gcp_vm_http_ok.sh
```

### Cannot connect via SSH
- Verify port 22 is allowed in the GCP firewall rules
- Confirm you are using the correct external IP
- Confirm your SSH key is attached to the VM

### VM external IP changed after restart
- Ephemeral IPs change every time a VM stops and starts
- Always get the current IP from GCP Console before running the gate script
- Consider reserving a static IP if you need a permanent address
